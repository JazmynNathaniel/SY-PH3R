# SY-PH3R Architecture Notes

## Trust Model

- Maximum of five approved members, each represented by one or more verified devices
- Membership starts with an out-of-band invite generated by an existing trusted operator
- Devices establish identity and verification state separately from the human member record
- The relay is not trusted with message plaintext
- The relay is trusted only to hold minimal metadata, encrypted envelopes, and delivery coordination state

## Threat Model Notes

- Primary goal: reduce interception and unauthorized access for a tiny private circle
- Non-goal: claiming the system is impossible to compromise
- Primary risks addressed in the MVP: public discovery, excessive metadata exposure, accidental plaintext storage, weak secret handling, and broad attack surface
- Risks deferred to later hardening: supply-chain review, advanced key recovery, deniable metadata protection, side-channel resistance, and formal cryptographic audit

## Layering

### Web Client

- Presentation layer for room, DMs, onboarding, and settings
- Domain layer for membership rules, message policy, and personalization constraints
- Storage layer abstraction for encrypted local drafts/cache
- Transport layer abstraction for relay sync and realtime envelope exchange
- Message encryption layer deriving a room key from a shared secret in the current MVP

### Relay API

- Auth/session boundary
- Invite/member/device lifecycle
- Relay-only room and encrypted envelope coordination
- Metadata minimization and retention controls

### Shared Package

- Domain limits such as maximum members
- Shared request/response shapes
- Personalization presets and screen-level contracts

## Storage Boundaries

- Relay database stores:
  - invite codes and expiry metadata
  - member records with minimal profile metadata
  - device fingerprints and verification state
  - session state
  - encrypted envelope headers and ciphertext blobs
- Relay database does not store:
  - plaintext message bodies
  - searchable public directories
  - analytics events
  - contact discovery data
- Browser local storage may store encrypted draft blobs only after client-side encryption

## Secure Defaults

- No public registration routes
- No password reset or email recovery flow
- No cloud backup by default
- No third-party analytics or ad SDKs
- Environment validation at process boot
- Explicit service interfaces for sensitive operations
- Draft persistence requires an explicit local unlock phrase in the current MVP
- Sensitive relay routes use bearer session tokens tied to approved devices

## Personalization Safety Model

- Profile photo uploads should be validated and transformed server-side in a hardened version
- Themes and layout choices are selected from predefined presets only
- Images are allowed as content/media in constrained flows
- No raw HTML, script execution, iframe embeds, or arbitrary CSS injection

## Suggested Hardening Path

1. Add audited E2EE envelope implementation and key lifecycle management.
2. Replace in-memory relay storage with SQLite migrations and retention controls.
3. Add authenticated WebSocket session management with replay protections.
4. Add device-verification UX backed by signed verification events.
5. Add privacy-preserving telemetry only if explicitly approved.
6. Remove the development-only bootstrap session route and replace it with a production operator-auth mechanism.
7. Replace the shared room secret model with per-member key distribution and forward-secrecy-aware messaging.
